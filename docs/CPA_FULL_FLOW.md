# CPA для товарного бизнеса: как должно работать

## 1) Эталонная цепочка событий

1. Affiliate получает уникальную трек-ссылку (`/t/:token`).
2. Покупатель кликает по ссылке.
3. Система фиксирует click: `token`, `subid`, `utm`, `ip`, `userAgent`, `timestamp`.
4. Пользователь оставляет заявку (lead) или оплачивает заказ (sale) на стороне поставщика.
5. Поставщик отправляет postback в `POST /api/events` с `external_id` (ID заказа/лида в CRM).
6. Система создаёт событие `lead`/`sale` со статусом `pending`.
7. Модерация (авто/ручная) переводит событие в `approved/rejected`.
8. При `approved` affiliate получает начисление.
9. Начисления попадают в реестр выплат и далее переводятся в `paid`.
10. Все стороны видят прозрачный статус, причину и следующий шаг.

## 2) Как это выглядит для покупателя и бизнеса

- Покупатель не видит внутреннюю механику CPA.
- Для покупателя путь простой: клик -> лендинг -> заявка/покупка.
- Для бизнеса критично, чтобы на каждом шаге была техническая фиксация:
  - кто привёл клиента (tracking token),
  - откуда трафик (source/subid/utm),
  - какая конверсия произошла (lead/sale),
  - какой итог статуса и почему.
